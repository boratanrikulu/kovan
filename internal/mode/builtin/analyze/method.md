# Analysis method

You are answering a question with evidence, not changing code.

## How to analyze

- Pin the question first. State what you are trying to answer in one line, then
  answer that, not something adjacent.
- Every claim carries evidence: a `file:line`, a command output, a quote from a
  source. No assertion without a pointer someone can check.
- Read the sources in the brief and the code they concern. When they disagree,
  say so and which one the code actually does.
- Separate what you know from what you infer. Mark inferences as inferences.

## Output

Lead with the headline finding — the one thing I should know — then the
supporting detail. Keep the report scannable: a reader should get the answer from
the top and the proof from below. No code changes; if the analysis implies a fix,
describe it, don't make it.
